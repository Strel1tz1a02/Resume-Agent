# 岗位与 JD 分析设计

## 目标

实现第四阶段“岗位与 JD 分析”，让用户仅粘贴 JD 原文即可创建岗位，并自动生成一份可人工修正的结构化 `JDAnalysis`。

本阶段使用简单、确定性的本地规则作为 Agent 占位实现。规则只负责跑通数据流，不追求高质量语义理解；后续接入 LLM 时替换服务层内部实现，不改变 API 和核心数据结构。

本阶段不实现岗位匹配、简历生成、自动投递或真实 LLM 调用。

## 核心流程

1. 用户在岗位页面粘贴 JD 原文并提交。
2. 前端调用 `POST /jobs` 创建 `JobPosting`。
3. 后端使用最小规则从 JD 中尽量填写公司、岗位名称、地点等岗位字段；无法识别的字段保留为空。
4. 岗位创建成功后，前端自动调用 `POST /jobs/{job_id}/jd-analyses`。
5. 后端创建新的 `JDAnalysis`，并将 `JobPosting.jd_analysis_id` 更新为该分析 ID。
6. 页面展示岗位详情和最新分析，用户可以人工修正两类数据。
7. 用户重新分析时，系统创建新的 `JDAnalysis`，保留旧记录，并更新岗位的当前分析指针。

岗位创建与 JD 分析是两个独立操作。首次导入时由前端自动串联，既保持使用体验连续，也允许分析失败后单独重试。

## 数据规则

### JobPosting

沿用现有字段：

- `company`
- `title`
- `location`
- `source_url`
- `raw_jd_text`
- `published_at`
- `deadline`
- `job_type`
- `status`
- `notes`
- `jd_analysis_id`

创建岗位时只要求 `raw_jd_text` 非空。其他字段可以由规则服务填写，也可以为空，并允许用户后续修改。

`raw_jd_text` 是原始输入和追溯来源。人工修改结构化字段时不覆盖原始 JD。

### JDAnalysis

沿用现有字段：

- `job_posting_id`
- `hard_requirements`
- `bonus_requirements`
- `keywords`
- `responsibilities`
- `capability_dimensions`
- `risks`
- `resume_emphasis`
- `completeness_status`

同一岗位可以拥有多条分析记录。`JobPosting.jd_analysis_id` 指向当前最新分析，历史分析通过 `job_posting_id` 保留。

分析结果允许人工编辑。编辑已有分析不会创建新版本；只有执行“重新分析”才创建新的记录。

## API 设计

### 岗位

- `GET /jobs`：列出当前用户的岗位，按 ID 排序。
- `POST /jobs`：根据 JD 原文创建岗位，并用最小规则尽量补全岗位字段。
- `GET /jobs/{job_id}`：读取岗位详情。
- `PUT /jobs/{job_id}`：人工修改岗位字段。
- `DELETE /jobs/{job_id}`：删除岗位。本阶段若岗位已有分析，删除行为同时删除其分析记录，避免遗留孤立数据。

`POST /jobs` 的最小请求体只需：

```json
{
  "raw_jd_text": "JD 原文"
}
```

同时允许请求体携带人工填写的其他岗位字段。人工传入的非空字段优先于规则提取结果。

### JD 分析

- `POST /jobs/{job_id}/jd-analyses`：基于岗位当前的 `raw_jd_text` 创建新分析，并更新岗位的 `jd_analysis_id`。
- `GET /jobs/{job_id}/jd-analyses`：列出该岗位的历史分析，最新记录在前。
- `GET /jd-analyses/{analysis_id}`：读取单条分析。
- `PUT /jd-analyses/{analysis_id}`：人工修改结构化分析。

创建分析时不要求请求体。规则服务始终读取指定岗位当前保存的 `raw_jd_text`。

岗位不存在或不属于当前用户时返回 `404`。JD 原文为空时不创建分析并返回 `422`。

## 规则服务

规则服务拆成两个独立函数：

- `extract_job_fields(raw_jd_text)`：尽量提取岗位基础信息。
- `analyze_jd(raw_jd_text)`：生成结构化分析字段。

规则实现保持简单，例如按行识别常见标签、按关键词和句子标记职责或要求、生成去重关键词，并根据可提取内容设置完整性状态。

规则不承担复杂语义推断，不伪造无法从原文确定的信息。服务输出必须使用与未来 LLM 服务相同的结构化结果类型，保证后续替换实现时不影响路由和前端。

完整性状态只使用：

- `complete`：至少识别出职责和要求。
- `incomplete`：缺少职责或要求。

## 前端交互

岗位页面沿用当前工作台布局：

- 左侧为岗位列表，顶部提供新增按钮。
- 中间主区域显示当前岗位详情和最新 `JDAnalysis`。
- 右侧 Agent 面板显示当前岗位上下文、分析状态和重新分析入口。

新增岗位通过弹窗完成，主要输入为 JD 原文，并允许用户同时填写或修正基础岗位字段。提交后页面自动执行创建和分析。

岗位详情中的基础字段可编辑保存。分析结果按字段分组展示，每个列表字段使用可编辑的逐行文本形式，完整性状态使用选择控件。

重新分析会明确提示将生成新的分析记录，不覆盖历史记录。历史分析本阶段通过简洁的版本选择器切换查看；默认显示 `JobPosting.jd_analysis_id` 指向的当前分析。

## 状态与错误处理

- 岗位列表加载失败：显示错误和重试操作。
- 岗位创建失败：关闭自动分析流程，不在前端加入临时岗位。
- 岗位创建成功但分析失败：保留并选中新岗位，显示“分析失败”和重新分析操作。
- 重新分析失败：保留当前分析，不修改 `jd_analysis_id`。
- 岗位或分析保存失败：保留用户当前编辑内容，允许再次保存。
- 切换岗位时：加载该岗位详情及其当前分析；没有分析时展示可触发分析的空状态。

## 组件边界

后端按现有结构增加：

- 岗位和分析 ORM 对应的 Pydantic schema。
- 岗位与分析路由。
- 独立的规则分析服务，路由不直接实现文本解析。

前端增加：

- 岗位和分析 API 客户端。
- 岗位列表、岗位编辑区、分析编辑区和新增岗位弹窗。
- 页面级状态仍沿用当前轻量实现，本阶段不引入新的状态管理或路由依赖。

若 `App.tsx` 因本阶段继续增长，应将岗位页面拆到独立组件中，但不对画像库和技能页面做无关重构。

## 测试范围

后端测试覆盖：

- 仅提供 JD 原文可以创建岗位。
- 人工提供的岗位字段优先于规则提取值。
- 创建分析后会回填 `JobPosting.jd_analysis_id`。
- 重复分析会保留历史并更新当前分析指针。
- 分析和岗位都可以人工修改。
- 空 JD 不能创建分析。
- 删除岗位会清理对应分析。
- 不同用户的数据不能通过 ID 越权读取或修改。

前端测试覆盖：

- 加载并选择岗位。
- 粘贴 JD 后自动执行创建和分析。
- 分析失败后仍保留已创建岗位并可重试。
- 编辑并保存岗位信息。
- 编辑并保存结构化分析。
- 重新分析后切换到最新分析，同时保留历史版本入口。

## 验收标准

- 用户只粘贴 JD 原文即可得到一条岗位记录和结构化分析。
- 用户可以修正岗位信息与分析结果。
- 同一岗位重复分析不会覆盖历史记录。
- 规则服务可在不修改 API、数据模型和前端调用方式的情况下替换为 LLM 实现。
- 岗位创建或分析失败时，已成功保存的数据不会丢失。
