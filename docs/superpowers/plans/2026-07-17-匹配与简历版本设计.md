# 匹配与简历版本设计

## 目标

实现第五阶段“匹配与简历版本”的完整数据链路，让用户可以从岗位的当前 JD 分析生成匹配报告，从报告候选材料中确认实际选材，再生成、编辑和预览独立保存的 Markdown 简历版本。

本阶段只验证产品流程、数据边界和交互，不设计真实匹配算法或高质量简历生成算法。后续替换服务层内部实现时，不修改 API、核心数据模型和前端调用流程。

本阶段不实现真实 LLM 调用、相关性排序、证据强弱判断、Word/PDF 导出、投递管理或自由对话。

## 核心流程

1. 用户在岗位页面选择一条已有 `JDAnalysis`。
2. 前端调用 `POST /jd-analyses/{analysis_id}/match` 创建新的 `MatchReport`。
3. 岗位页面展示匹配报告，并允许在同一分析的历史报告之间切换。
4. 用户点击“生成简历”，在候选经历和技能中确认实际采用的内容。候选内容默认全选，用户可以取消或恢复选择。
5. 前端调用 `POST /match-reports/{report_id}/resume-versions`，提交最终选择的 Experience 和 Skill ID。
6. 后端创建 `ResumeVersion`，保存 Markdown、实际采用的材料 ID 和生成说明。
7. 前端切换到独立的“简历版本”页面，并选中新生成的版本。
8. 用户编辑 Markdown、即时预览并保存。保存不会修改原始画像事实或 `MatchReport`。

## 数据边界

### MatchReport

`MatchReport` 表示一次匹配结果快照，沿用现有字段：

- `jd_analysis_id`
- `overall_score`
- `candidate_experience_ids`
- `candidate_skill_ids`
- `matched_requirements`
- `gaps`
- `risks`
- `follow_up_questions`
- `resume_strategy`

同一 `JDAnalysis` 可以生成多条报告。新报告不会覆盖历史报告。候选 Experience/Skill ID 只表示可供简历选材的内容，不表示简历最终实际采用了这些内容。

### ResumeVersion

`ResumeVersion` 表示一次简历内容快照，沿用现有字段：

- `job_posting_id`
- `match_report_id`
- `markdown_content`
- `used_experience_ids`
- `used_skill_ids`
- `generation_rationale`
- `manual_edit_history`
- 创建和更新时间

`used_experience_ids` 和 `used_skill_ids` 只记录用户生成时最终确认的材料。它们必须是对应 `MatchReport` 候选 ID 的子集，并与候选字段保持明确区分。

`job_posting_id` 由后端通过 `MatchReport -> JDAnalysis -> JobPosting` 在创建版本时确定，前端不提交该字段。

## 占位服务

### 匹配占位服务

本阶段的匹配服务只跑通链路：

- 读取指定 `JDAnalysis`。
- 读取当前用户的全部 Experience 和 Skill。
- 将全部 Experience/Skill ID 分别写入候选 ID 字段。
- 使用固定、可预测的占位分数和占位说明填充报告其余字段。
- 当画像库没有经历或技能时仍创建报告，并通过缺口或追问字段提示补充画像。

服务不进行关键词计算、相关性打分、排序或语义推断。

### 简历生成占位服务

简历生成服务只读取：

- `MatchReport`
- 用户最终选择的 Experience
- 用户最终选择的 Skill

它不得读取 `JobPosting` 或 `JDAnalysis`。路由可以在调用服务前通过关系链确定并保存 `job_posting_id`，但岗位和分析内容不得进入简历内容生成服务。

服务使用固定 Markdown 模板拼接用户选择的事实内容，不制造新的事实、指标或成果。`generation_rationale` 使用明确的占位说明，便于后续替换真实 Agent。

## API 设计

### 匹配报告

- `POST /jd-analyses/{analysis_id}/match`：创建一条新报告。
- `GET /jd-analyses/{analysis_id}/match-reports`：按最新在前列出该分析的报告历史。
- `GET /match-reports/{report_id}`：读取报告及候选材料展示数据。

岗位或分析不存在、或不属于当前用户时返回 `404`。

### 简历版本

- `POST /match-reports/{report_id}/resume-versions`：根据最终选择的候选材料创建版本。
- `GET /resume-versions`：按最新在前列出当前用户的全部版本，并返回岗位展示信息。
- `GET /resume-versions/{version_id}`：读取版本和实际采用的材料展示数据。
- `PUT /resume-versions/{version_id}`：保存用户编辑后的 Markdown。

创建请求只提交：

```json
{
  "used_experience_ids": [1, 2],
  "used_skill_ids": [3]
}
```

提交的 ID 必须属于报告候选集合、属于当前用户且不存在重复值。允许任一列表为空。非法选择返回 `422`，不创建部分版本。

更新接口本阶段只允许更新 `markdown_content`。保存成功时，后端向 `manual_edit_history` 追加包含修改时间和修改前后摘要的记录。更新操作不改变 `used_experience_ids`、`used_skill_ids` 或原报告。

## 前端交互

### 岗位页面

岗位页面继续保留岗位列表和 JD 分析编辑区，并增加匹配报告区域：

- 当前分析存在时显示“生成匹配报告”。
- 创建报告后默认显示新报告。
- 提供简洁的历史报告选择器。
- 展示匹配分、匹配要求、缺口、风险、追问、策略和候选材料。
- 候选经历和技能显示可辨认的名称、类型或分类，但不标记为已经用于简历。
- “生成简历”打开选材弹窗，候选项默认全选，并允许用户调整。
- 创建成功后切换到“简历版本”导航并选中新版本。

匹配创建失败时保留当前 JD 分析和已有报告。简历创建失败时保留弹窗中的用户选择，允许重试。

### 简历版本页面

独立页面包含：

- 左侧版本列表，显示目标公司、岗位和创建时间。
- 中间 Markdown 编辑器和即时预览。
- 当前版本实际采用的经历与技能列表。
- 保存按钮和明确的保存状态。

第一版预览只支持占位模板实际使用的标题、段落和列表等基础 Markdown，不增加第三方富文本编辑器或复杂 Markdown 依赖。

保存失败时保留编辑内容并允许重试。切换版本时，迟到请求不得覆盖当前选中的版本。

### Agent 面板

- 岗位页解释当前显示的是候选材料，并提示用户在生成简历前确认选材。
- 简历页说明当前实际采用的材料和占位生成边界。
- 本阶段不加入真实聊天能力。

## 组件边界

后端新增：

- 匹配报告与简历版本 Pydantic schema。
- 匹配报告和简历版本路由。
- 独立匹配占位服务。
- 独立简历生成占位服务。

前端新增：

- 匹配报告和简历版本 API 客户端。
- 岗位页中的匹配报告区域和选材弹窗。
- 独立简历版本页面。
- 仅覆盖基础 Markdown 的安全预览组件。

前端沿用现有页面级轻量状态和请求隔离方式，本阶段不引入新的路由或全局状态依赖。

## 错误与权限处理

- 所有查询和写入都限定当前用户。
- 跨用户或不存在的分析、报告、版本和材料统一按资源不可见处理。
- 无 JDAnalysis 时不能生成报告。
- 选择非候选材料、重复 ID 或其他用户材料时不创建版本。
- 匹配或生成失败不覆盖已有记录。
- Markdown 保存失败不丢失前端草稿。
- 前端切换岗位、分析、报告或版本时，旧请求结果不得覆盖当前选择。

## 测试范围

后端测试覆盖：

- 从已有 JDAnalysis 创建 MatchReport。
- 占位报告将当前用户 Experience/Skill 记录为候选。
- 空画像仍能创建报告。
- 重复生成报告会保留历史。
- 不同用户不能读取或基于他人分析生成报告。
- 从报告候选集合的子集生成 ResumeVersion。
- 候选 ID 与实际使用 ID 保持区别。
- 非候选、重复或跨用户材料不能用于生成版本。
- 简历生成服务只接收 MatchReport、Experience 和 Skill，不读取 JobPosting/JDAnalysis。
- Markdown 更新会追加人工修改历史且不改变选材 ID。

前端测试覆盖：

- 当前分析可以生成并显示匹配报告。
- 新报告生成后成为当前报告，历史报告仍可切换。
- 生成简历弹窗默认全选候选材料并允许调整。
- 生成成功后进入简历版本页面并选中新版本。
- 简历列表、实际采用材料、Markdown 编辑和预览正确显示。
- 保存失败时保留 Markdown 草稿并可重试。
- 迟到请求不会覆盖当前选择。

## 验收标准

- 用户可以从岗位当前 JD 分析生成一条可查看的 MatchReport。
- 用户可以从候选材料中确认实际选材并生成 ResumeVersion。
- ResumeVersion 明确记录实际使用的 Experience/Skill ID，且它们是报告候选 ID 的子集。
- 用户可以在独立简历版本页面编辑、预览和保存 Markdown。
- 候选内容与实际使用内容在数据和界面上都有清晰区别。
- 占位服务可以在不修改 API、核心数据模型和前端流程的情况下替换为真实算法或 LLM 服务。
