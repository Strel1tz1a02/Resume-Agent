# 投递、Markdown 导出与应用配置设计

## 目标

实现第六阶段“投递与导出占位”，完成 `ResumeVersion -> ApplicationRecord` 的最后一段 MVP 数据链路，并提供 Markdown 浏览器下载和本地应用配置页面。

本阶段把投递记录、Markdown 导出和应用配置实现为三个独立纵向闭环。它们共享现有工作台和用户归属规则，但不通过单个聚合接口相互耦合。

本阶段不实现自动投递、后端文件持久化、指定浏览器下载目录、Word/PDF 渲染、真实模型调用、数据目录迁移或云同步。

## 核心原则

- SQLite 中的 `ResumeVersion.markdown_content` 是简历内容的唯一正式版本。
- Markdown 下载是从数据库内容即时生成的浏览器下载副本，不在后端长期保存文件。
- 浏览器决定下载目录，应用不配置也不承诺本地下载路径。
- `AppConfig.default_output_path` 保留为废弃兼容字段，但不在本阶段 API 或 UI 中暴露。
- 投递模块只追踪状态，不自动向招聘网站提交任何内容。
- ApplicationRecord 创建后，其岗位和简历关联不可修改。

## 模块一：投递记录

### 数据规则

沿用现有 `ApplicationRecord` 字段：

- `job_posting_id`
- `resume_version_id`
- `status`
- `applied_at`
- `result`
- 创建和更新时间

固定状态只允许：

- `preparing`：准备中
- `applied`：已投递
- `written_test`：笔试
- `interview`：面试
- `offer`：Offer
- `rejected`：已拒绝
- `withdrawn`：已撤回

新建记录默认状态为 `preparing`。`result` 保留为自由文本，用于记录“二面未通过”“等待 HR 确认”等具体信息。

同一岗位第一版最多存在一条 ApplicationRecord。重复创建返回冲突，不通过不同简历版本绕过此约束。创建时选择的 ResumeVersion 必须属于同一个 JobPosting。

投递记录创建后，只允许更新 `status`、`applied_at` 和 `result`；不允许更换岗位或简历版本。

### API

- `GET /applications`：按最新在前列出当前用户的投递记录。
- `POST /applications`：创建投递记录。
- `PUT /applications/{application_id}`：更新状态、投递时间和结果。

创建请求：

```json
{
  "job_posting_id": 7,
  "resume_version_id": 31,
  "status": "preparing",
  "applied_at": null,
  "result": null
}
```

列表和更新响应除 ORM 字段外，还返回前端直接展示所需的：

- `job_company`
- `job_title`
- `resume_version_label`

岗位、简历或投递不存在或不属于当前用户时返回 `404`。简历不属于指定岗位或状态非法时返回 `422`。同一岗位已有投递记录时返回 `409`。

### 前端交互

“投递清单”页面显示：

- 公司和岗位
- 使用的简历版本
- 投递状态
- 投递时间
- 结果

页面顶部提供“新增投递”。创建弹窗先选择岗位，再只显示属于该岗位的 ResumeVersion。没有可用简历时明确提示用户先生成简历版本。

点击已有记录打开编辑弹窗。编辑时岗位和简历只读，只能修改状态、投递时间和结果。

当用户第一次把状态改为 `applied` 且投递时间为空时，前端默认填入当前本地时间。用户可以修改该时间；后续切换状态不会自动覆盖已有值。

右侧 Agent 面板明确提示本页面只追踪投递，不会自动提交。

## 模块二：Markdown 浏览器下载

### 数据与文件边界

后端不维护长期存在的 `.md` 文件。每次导出都执行：

1. 按当前用户读取 ResumeVersion。
2. 读取该版本数据库中的 `markdown_content`。
3. 根据公司、岗位和版本号生成安全文件名。
4. 返回带 `Content-Disposition: attachment` 的 UTF-8 Markdown 响应。
5. 浏览器决定保存位置。

修改下载后的文件不会同步回应用；在应用中保存新的 Markdown 也不会覆盖过去下载的副本。

### API

- `POST /resume-versions/{version_id}/export`

本阶段不需要请求体，响应媒体类型为 `text/markdown; charset=utf-8`。

建议文件名格式：

```text
公司_岗位_简历版本31.md
```

文件名服务必须清理 Windows 和常见文件系统不允许的字符，并为缺失公司或岗位名称提供稳定后备值。

导出不存在或不属于当前用户的版本时返回 `404`。导出不新增数据库记录，不修改 ResumeVersion，也不读取或写入 `default_output_path`。

### 前端交互

“简历版本”页面增加“下载 Markdown”按钮。

如果当前 Markdown 草稿与已保存版本不同，按钮禁用并提示“请先保存后下载”，避免用户误以为下载包含未保存内容。保存成功后按钮恢复可用。

前端调用导出接口，读取 Blob 和响应文件名，再触发浏览器下载。下载失败时保留当前页面、选中版本和编辑草稿，显示可重试错误。

## 模块三：应用配置

### 配置边界

沿用 AppConfig 单例记录，但本阶段 API 只暴露：

- `resume_template`
- `preferred_export_formats`
- `model_provider`
- `model_config`
- `language_preference`
- `data_directory`
- `privacy_settings`

`default_output_path` 不出现在请求 schema、响应 schema 或前端页面中。数据库字段暂时保留，以避免为未使用字段新增删除迁移；后续统一整理数据库时再移除。

当前唯一可执行的导出格式是 `markdown`。`preferred_export_formats` 本阶段只允许空列表或包含 `markdown`，不提供不可用的 Word/PDF 选项。

模型配置只保存占位设置，不触发模型连接、密钥验证或模型调用。数据目录只展示，不通过本页面移动 SQLite 数据库或附件。

### API

- `GET /app-config`：读取当前用户配置；尚未保存时返回带稳定默认值的配置视图。
- `PUT /app-config`：创建或更新当前用户唯一配置记录。

PUT 使用完整表单语义。首次保存时创建记录，之后更新同一记录，不产生历史版本。

请求中的 `model_config` 和 `privacy_settings` 必须是 JSON 对象，`preferred_export_formats` 必须符合当前支持范围。跨用户记录不可见。

### 前端交互

配置页面使用单页表单：

- 简历模板：普通输入。
- 导出格式：展示“Markdown（当前支持）”。
- 模型供应商：普通输入。
- 模型配置：JSON 文本框。
- 语言偏好：选择控件。
- 数据目录：只读展示。
- 隐私设置：JSON 文本框。

模型配置和隐私设置在前端保存前解析；只有 JSON 对象可以提交。解析失败时不发送请求，保留原始输入并显示字段错误。

右侧 Agent 面板说明模型设置当前只是本地保存占位，不会触发真实模型调用。

## 组件边界

后端新增：

- ApplicationRecord schema 与路由。
- AppConfig schema 与路由。
- Markdown 导出服务和导出路由。
- 文件名清理纯函数。

投递路由负责权限、岗位与简历一致性、唯一投递约束和持久化。导出服务只负责把已保存内容与展示元数据转成下载响应所需结果。配置路由负责当前用户单例 upsert。

前端新增：

- 投递和配置 API 客户端。
- 独立投递清单页面。
- 独立应用配置页面。
- 简历 API 中的导出函数，以及简历页下载交互。

页面继续沿用现有轻量状态和请求 epoch 模式，不引入 React Router、TanStack Query、Zustand 或新的 UI 框架。

## 一致性与错误处理

- 所有业务查询和写入限定当前用户。
- ApplicationRecord 的 JobPosting 与 ResumeVersion 必须匹配。
- 投递关联创建后不可变。
- 同一岗位重复创建投递返回 `409`。
- 非法状态返回 `422`。
- 导出始终读取数据库已保存内容；存在未保存前端草稿时不允许发起导出。
- 导出失败不改变数据库，不丢失前端草稿。
- AppConfig JSON 字段无效时不提交；保存失败保留表单内容。
- 投递或配置列表加载失败显示重试入口。
- 页面或记录切换时，迟到请求不得覆盖当前选择。

## 测试范围

### 后端

- 创建 ApplicationRecord 会校验岗位和简历属于当前用户。
- ResumeVersion 与 JobPosting 不一致时不能创建投递。
- 同一岗位重复创建投递返回 `409`。
- 只允许固定投递状态。
- 更新状态、时间和结果不会改变岗位或简历关联。
- 跨用户不能读取、创建或修改投递。
- 导出响应内容等于数据库里的 `markdown_content`。
- 导出文件名合法、响应类型正确，且不在后端创建文件。
- 导出不存在或跨用户简历返回 `404`。
- AppConfig GET 返回默认视图，PUT 首次创建、再次更新同一记录。
- AppConfig API 不暴露或接受 `default_output_path`。
- AppConfig 拒绝不支持的导出格式和非对象 JSON 字段。

### 前端

- 投递清单可加载、新建和编辑。
- 创建时只显示所选岗位对应的简历版本。
- 第一次切换到已投递状态会在空时间字段中填入当前本地时间。
- 投递创建或更新失败时保留弹窗内容并可重试。
- 简历有未保存草稿时禁止下载。
- 保存后可以下载 Markdown，并采用服务器响应文件名。
- 导出失败时保留当前简历草稿。
- 配置页面能加载和保存各字段。
- 配置 JSON 无效时不发送请求并保留输入。
- 配置保存失败时保留表单并可重试。
- 迟到请求不会覆盖当前投递或配置状态。

## 验收标准

- 用户可以选择岗位及其简历版本加入投递清单。
- 用户可以更新投递状态、投递时间和结果。
- 同一岗位不会产生重复投递记录，且岗位与简历始终一致。
- 用户可以从已保存 ResumeVersion 下载内容一致的 Markdown 副本。
- 浏览器决定下载路径，应用不显示或使用输出路径配置。
- 用户可以读取和保存 AppConfig，模型和隐私 JSON 配置经过校验。
- `default_output_path` 不在本阶段 API 或 UI 出现。
- 三个模块可以独立测试和替换，不互相依赖内部实现。
